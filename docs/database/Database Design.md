## 一、文档目的

这份文档用于解释 DeveloperOS V1 数据库为什么这样设计，以及后端开发时应该怎样正确使用这些表。

文档主要回答：

- PostgreSQL 中表、主键、外键、约束、索引和事务是什么；
- V1 为什么选择当前 9 张核心表；
- 每张表及每个字段的作用是什么；
- 表之间如何关联，删除数据时会发生什么；
- AI 对话、长期记忆、RAG 和成长系统如何读写数据；
- 哪些字段由数据库维护，哪些必须由 Spring Boot Service 维护；
- 后续如何使用 Flyway 安全演进数据库；
- V2、V3、V4 可以沿什么方向扩展。

`V1__init_schema.sql` 是当前正式的 Flyway 初始迁移，本文档是对该结构的解释。以后数据库结构发生变化时，应新增迁移脚本，再同步更新本文档。

## 二、数据库在 DeveloperOS 中的职责

DeveloperOS 不会把所有数据都存进 PostgreSQL，不同组件承担不同职责。

| 组件 | 主要职责 | 典型数据 |
| --- | --- | --- |
| PostgreSQL | 保存需要长期存在、需要事务和关系约束的业务数据 | 用户、会话、消息、记忆、目标、任务、日志、文档元数据 |
| pgvector | PostgreSQL 扩展，用于保存和检索向量 | 知识切片 Embedding |
| Redis | 保存短期、高频或有过期时间的数据 | Token 状态、验证码、缓存、临时上下文、限流计数 |
| 文件系统或对象存储 | 保存文件本体 | PDF、Markdown、Word、文本文件 |
| LLM 服务 | 生成回答、提取记忆、生成向量 | 不作为业务数据最终存储位置 |

重要边界：

- `knowledge_document` 只保存文件元数据和 `storage_key`，不保存整个文件；
- Redis 数据可以过期，PostgreSQL 中的成长数据需要长期保存；
- pgvector 不是独立数据库，而是 PostgreSQL 扩展；
- JWT 登录状态按照系统架构放在 Redis，V1 不需要 Token 表。

## 三、第一次使用 PostgreSQL 需要理解的概念

### 3.1 Database、Schema、表、行和列

| 概念 | 含义 | DeveloperOS 示例 |
| --- | --- | --- |
| Database | 一个独立数据库 | `developeros` |
| Schema | 数据库内部命名空间 | 当前使用默认 `public` Schema |
| Table | 保存同一类数据的结构 | `app_user`、`conversation` |
| Row | 一条具体记录 | 一个用户、一条消息 |
| Column | 记录中的一个字段 | `username`、`created_at` |

当前项目已经创建 `developeros` 数据库。初始化脚本只在该数据库中创建扩展、函数、表、索引和触发器，不执行 `CREATE DATABASE`。

### 3.2 主键

主键唯一标识一条记录：

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

- `BIGINT` 在 Java 中通常映射为 `Long`；
- `IDENTITY` 表示默认由 PostgreSQL 自动生成递增 ID；
- `PRIMARY KEY` 保证 ID 非空且唯一，并自动创建索引。

使用 MyBatis-Plus 时，应将实体 ID 策略设置为数据库自增，例如 `IdType.AUTO`，避免和默认雪花 ID 冲突。

### 3.3 外键

外键保证子表引用的父记录真实存在：

```sql
FOREIGN KEY (conversation_id)
REFERENCES conversation (id)
ON DELETE CASCADE
```

这表示消息必须属于真实会话，删除会话时其消息一起删除。

外键保证数据关系，不等于接口权限。后端仍必须验证当前登录用户是否拥有对应会话、文档、目标或日志。

### 3.4 索引

索引类似书的目录，可以提高查询速度，但会占用空间并增加写入成本。

```sql
CREATE INDEX idx_conversation_user_updated
ON conversation (user_id, updated_at DESC);
```

这个索引服务于“查询某个用户最近会话”的真实场景。不能给每个字段都加索引，应等接口和查询形成后通过 `EXPLAIN ANALYZE` 判断。

### 3.5 约束

本项目使用：

- `NOT NULL`：字段不能为空；
- `UNIQUE`：字段或字段组合不能重复；
- `CHECK`：字段值必须满足规则；
- `FOREIGN KEY`：引用记录必须存在；
- `PRIMARY KEY`：记录唯一标识。

应用层校验负责友好错误信息，数据库约束负责最后一道数据保护，两者不能互相替代。

### 3.6 事务

事务保证一组操作要么全部成功，要么全部失败。

例如保存用户消息时，需要同时：

1. 插入 `message`；
2. 更新 `conversation.last_message_at`；
3. 更新 `conversation.updated_at`。

这些操作应放在同一个 Spring `@Transactional` 方法中，防止只保存了消息却没有更新会话时间。

### 3.7 常用字段类型

| PostgreSQL 类型 | Java 建议类型 | 用途 |
| --- | --- | --- |
| `BIGINT` | `Long` | 主键、外键、文件大小 |
| `INTEGER` | `Integer` | Token 数、排序值、切片数量 |
| `SMALLINT` | `Integer` 或 `Short` | 优先级、百分比 |
| `VARCHAR(n)` | `String` | 标题、状态、名称 |
| `TEXT` | `String` | 消息、记忆、日志等长文本 |
| `BOOLEAN` | `Boolean` | 是否由 AI 生成 |
| `DATE` | `LocalDate` | 学习日期、目标日期 |
| `TIMESTAMPTZ` | `Instant` 或 `OffsetDateTime` | 带时区语义的时间点 |
| `JSONB` | `JsonNode`、Map 或专用对象 | 可扩展结构化数据 |
| `VECTOR` | pgvector 对应类型 | Embedding 向量 |

项目应统一时间类型和时区策略。推荐数据库保存绝对时间，前端展示时再转换为用户时区。

## 四、V1 设计原则

### 4.1 面向 MVP

V1 只创建支撑用户、AI 对话、长期记忆、RAG、学习目标和成长记录的表。提醒、多 Agent、GitHub、社区和团队协作不属于当前范围。

### 4.2 模块化单体

当前使用一个 PostgreSQL 数据库，按照业务模块划分表。相比一开始拆微服务，这更适合个人项目和 MVP，也保留了后续拆分空间。

### 4.3 用户数据隔离

每个业务根表都能追溯到 `app_user`：

- 会话：`conversation.user_id`；
- 记忆：`memory.user_id`；
- 文档：`knowledge_document.user_id`；
- 目标：`learning_goal.user_id`；
- 日志：`growth_log.user_id`。

子表通过父表追溯用户，例如 `message -> conversation -> app_user`。

### 4.4 核心字段结构化，扩展字段使用 JSONB

需要查询、排序、关联或约束的字段使用普通列。暂时没有稳定结构、只用于附加信息的字段放入 `metadata` 或 `preferences`。

不能为了省事把所有字段都放入 JSONB，否则查询、索引、校验和 Java 映射会变得困难。

### 4.5 当前使用物理删除

V1 没有给每张表增加 `deleted` 或 `deleted_at`：

- MVP 阶段减少所有查询都添加删除条件的复杂度；
- 用户明确删除账户时，应能彻底删除个人数据；
- 普通非活跃状态通过 `archived`、`cancelled`、`disabled` 表达。

后端不能把停用账户实现成删除用户。正常停用只修改 `app_user.status`，只有明确的数据删除操作才允许物理删除。

## 五、核心表和关系总览

### 5.1 V1 核心表

| 模块 | 表名 | 作用 |
| --- | --- | --- |
| 用户 | `app_user` | 账号、资料、偏好和状态 |
| 聊天 | `conversation` | AI 会话 |
| 聊天 | `message` | 会话消息 |
| 记忆 | `memory` | 用户长期信息 |
| 知识库 | `knowledge_document` | 文件元数据和处理状态 |

| 成长 | `learning_goal` | 学习目标 |
| 成长 | `learning_task` | 目标下的具体任务 |
| 成长 | `growth_log` | 每日、每周、里程碑和复盘记录 |

### 5.2 ER 关系图

```text
app_user
 ├── conversation
 │      └── message
 │
 ├── memory
 │
 ├── knowledge_document
 │      └── knowledge_chunk
 │
 ├── learning_goal
 │      ├── learning_task
 │      └── growth_log（可选关联）
 │
 └── growth_log
```

- 一个用户可以有多个会话、记忆、文档、目标和日志；
- 一个会话有多条消息；
- 一个文档有多个知识切片；
- 一个目标有多个学习任务；
- 成长日志必须属于用户，也可以选择关联该用户的目标。

## 六、命名和通用字段

### 6.1 命名规则

- 表名使用小写单数形式；
- 字段使用 `snake_case`；
- 主键统一为 `id`；
- 外键使用 `<语义>_id`；
- 索引使用 `idx_`；
- 唯一约束使用 `uk_`；
- 外键使用 `fk_`；
- CHECK 约束使用 `ck_`；
- 触发器使用 `trg_`。

用户表不使用 `user`，因为 `USER` 在 PostgreSQL 中具有关键字和当前用户语义。`app_user` 可以避免 SQL 加双引号和 ORM 映射问题。

### 6.2 通用时间字段

大多数表有：

| 字段 | 作用 |
| --- | --- |
| `created_at` | 第一次创建时间 |
| `updated_at` | 最后一次修改时间 |

脚本创建 `set_updated_at()` 函数，并为 9 张表建立 `BEFORE UPDATE` 触发器。后端更新记录时不需要手动传递 `updated_at`。

## 七、用户表 `app_user`

### 7.1 职责

保存登录账号、基本资料、偏好和账号状态，是所有用户数据的根表。

### 7.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键，自动生成 | 用户唯一 ID |
| `username` | `VARCHAR(50)` | 必填 | 登录名或唯一用户名 |
| `email` | `VARCHAR(255)` | 必填 | 用户邮箱 |
| `password_hash` | `VARCHAR(255)` | 必填 | BCrypt、Argon2 等算法生成的哈希，绝不能保存明文密码 |
| `nickname` | `VARCHAR(100)` | 可空 | 页面展示昵称 |
| `avatar_url` | `TEXT` | 可空 | 头像地址 |
| `bio` | `VARCHAR(500)` | 可空 | 用户简介 |
| `timezone` | `VARCHAR(64)` | 默认 `Asia/Shanghai` | 每日任务、日志和提醒使用的时区 |
| `preferences` | `JSONB` | 默认空对象 | 回答风格、界面设置等可扩展偏好 |
| `status` | `VARCHAR(20)` | 默认 `active` | 账号状态 |
| `last_login_at` | `TIMESTAMPTZ` | 可空 | 最近成功登录时间 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 注册时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 资料最后修改时间 |

### 7.3 状态

| 状态 | 含义 |
| --- | --- |
| `active` | 正常使用 |
| `disabled` | 主动或管理员停用 |
| `locked` | 因安全原因锁定 |

### 7.4 约束和索引

- 用户名去除首尾空格后的长度至少为 3；
- 邮箱不能为空字符串；
- `lower(username)` 唯一，避免 `Keyx` 和 `keyx` 成为两个用户；
- `lower(email)` 唯一，避免邮箱大小写造成重复；
- 邮箱格式由 DTO `@Email` 和 Service 校验；
- Service 保存前应 `trim`，邮箱建议转为小写。

### 7.5 删除影响

删除用户会级联删除其会话、消息、记忆、文档、切片、目标、任务和日志。因此普通停用必须修改状态，删除用户只能用于明确的数据清除。

## 八、会话表 `conversation`

### 8.1 职责

表示一次 AI 对话，用于历史会话、多轮上下文和模型归属。

### 8.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 会话唯一 ID |
| `user_id` | `BIGINT` | 必填，外键 | 所属用户 |
| `title` | `VARCHAR(200)` | 默认 `New conversation` | 会话列表标题，可由首条消息或 AI 生成 |
| `model_name` | `VARCHAR(100)` | 可空 | 会话默认模型 |
| `status` | `VARCHAR(20)` | 默认 `active` | 正常或归档状态 |
| `last_message_at` | `TIMESTAMPTZ` | 可空 | 最后一条消息时间 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 最后修改时间 |

状态只有 `active` 和 `archived`。

`last_message_at` 是派生字段。`MessageService` 保存消息时必须在同一事务更新它，否则最近会话排序会错误。

索引：

- `(user_id, updated_at DESC)` 支持最近会话列表；
- `(user_id, status)` 支持正常和归档列表。

删除会话会级联删除消息，但不会删除已经沉淀的长期记忆。

## 九、消息表 `message`

### 9.1 职责

保存每条用户、系统、AI 或工具消息，支撑历史消息、多轮上下文、SSE 状态和错误记录。

### 9.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 消息唯一 ID |
| `conversation_id` | `BIGINT` | 必填，外键 | 所属会话 |
| `role` | `VARCHAR(20)` | 必填 | `system`、`user`、`assistant`、`tool` |
| `content` | `TEXT` | 默认空字符串 | 消息文本，流式消息创建时可以暂时为空 |
| `status` | `VARCHAR(20)` | 默认 `completed` | 消息生成状态 |
| `model_name` | `VARCHAR(100)` | 可空 | 实际生成模型 |
| `token_count` | `INTEGER` | 可空 | Token 数量或模型使用量 |
| `error_message` | `TEXT` | 可空 | 失败原因摘要 |
| `metadata` | `JSONB` | 默认空对象 | 工具调用、模型参数、引用来源等 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 内容或状态最后更新时间 |

状态：

| 状态 | 含义 |
| --- | --- |
| `streaming` | 正在流式生成 |
| `completed` | 生成完成 |
| `failed` | 生成失败 |
| `stopped` | 用户或系统中止 |

索引 `(conversation_id, created_at, id)` 支持稳定读取消息顺序，V1 不需要额外的 `sequence_no`。

消息没有重复保存 `user_id`，用户归属为：

```text
message -> conversation -> conversation.user_id
```

接口不能只按消息 ID 返回数据，必须校验会话属于当前用户。

## 十、长期记忆表 `memory`

### 10.1 职责

保存 AI 从对话、日志或手动输入中沉淀出的长期信息。

- `message` 保存用户说过什么；
- `memory` 保存未来仍有价值的稳定信息。

### 10.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 记忆唯一 ID |
| `user_id` | `BIGINT` | 必填，外键 | 所属用户 |
| `memory_type` | `VARCHAR(30)` | 必填 | 记忆分类 |
| `content` | `TEXT` | 必填 | 整理后的记忆内容 |
| `source_type` | `VARCHAR(30)` | 默认 `manual` | 来源类型 |
| `source_id` | `BIGINT` | 可空 | 来源记录 ID，由 `source_type` 解释 |
| `importance` | `SMALLINT` | 默认 3 | 重要度 1 到 5 |
| `status` | `VARCHAR(20)` | 默认 `active` | 是否仍然有效 |
| `last_accessed_at` | `TIMESTAMPTZ` | 可空 | 最近被检索使用的时间 |
| `metadata` | `JSONB` | 默认空对象 | 置信度、标签、来源片段等 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 最后修正时间 |

记忆类型：

| 类型 | 示例 |
| --- | --- |
| `goal` | 三个月掌握 Spring Security |
| `preference` | 喜欢先解释原因再给代码 |
| `profile` | Java 后端方向的软件工程学生 |
| `project` | 正在开发 DeveloperOS |
| `experience` | 完成过 JWT 登录模块 |
| `event` | 完成一次面试或里程碑 |
| `other` | 暂时无法归类 |

来源类型为 `manual`、`conversation`、`growth_log`、`system`。

`source_id` 是多态引用，数据库无法建立普通外键。Service 必须按 `source_type` 校验对应记录且属于同一用户。

有效记忆使用 `active`，过时或被替代的记忆改为 `archived`。记忆提取必须做去重、冲突检查和用户纠正。V1 暂不保存记忆向量，后续可新增向量列或 `memory_embedding` 表。

## 十一、知识文档表 `knowledge_document`

### 11.1 职责

保存上传文档的元数据、存储位置和处理状态。文件本体存放在本地文件目录、MinIO、S3 等位置。

### 11.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 文档唯一 ID |
| `user_id` | `BIGINT` | 必填，外键 | 所属用户 |
| `title` | `VARCHAR(255)` | 必填 | 知识库展示标题 |
| `original_name` | `VARCHAR(255)` | 必填 | 上传时原文件名 |
| `storage_key` | `VARCHAR(500)` | 必填且唯一 | 文件系统或对象存储中的唯一标识 |
| `mime_type` | `VARCHAR(150)` | 可空 | 如 `application/pdf` |
| `file_size` | `BIGINT` | 必填 | 文件字节数，不能小于 0 |
| `checksum_sha256` | `CHAR(64)` | 可空 | 内容校验和重复检测 |
| `status` | `VARCHAR(20)` | 默认 `pending` | 文档处理状态 |
| `chunk_count` | `INTEGER` | 默认 0 | 切片数量 |
| `error_message` | `TEXT` | 可空 | 处理失败原因 |
| `metadata` | `JSONB` | 默认空对象 | 页数、语言、解析器版本等 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 上传记录创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 状态最后更新时间 |

状态机：

```text
pending -> processing -> ready
                    -> failed
failed  -> processing（重试）
```

`chunk_count` 是派生字段。切片处理完成后，Knowledge Service 应在事务中同时完成：替换切片、更新数量、清理错误、更新状态。

文件安全规则：

- 上传、下载、重建向量和删除前校验 `user_id`；
- `storage_key` 由后端生成，不能直接使用用户路径；
- 原始文件名只用于展示，不能直接拼接服务器路径；
- 文件类型、大小和解析超时必须限制；
- 删除文档时切片由数据库级联删除，实际文件由 Service 删除。

## 十二、知识切片表 `knowledge_chunk`

### 12.1 职责

保存文档切片文本及其 Embedding，是 RAG 检索核心表。

### 12.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 切片唯一 ID |
| `document_id` | `BIGINT` | 必填，外键 | 所属文档 |
| `chunk_index` | `INTEGER` | 必填 | 文档内顺序，从 0 开始 |
| `content` | `TEXT` | 必填 | 切片文本 |
| `token_count` | `INTEGER` | 可空 | Token 数，用于控制上下文 |
| `embedding` | `VECTOR` | 可空 | 文本向量 |
| `embedding_model` | `VARCHAR(100)` | 可空 | 生成向量的模型 |
| `metadata` | `JSONB` | 默认空对象 | 页码、章节、字符范围等 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 最后更新时间 |

`UNIQUE (document_id, chunk_index)` 保证同一文档不会有两个相同序号，并自动创建索引，不需要字段相同的普通索引。

切片通过 `knowledge_chunk -> knowledge_document -> user_id` 确定用户。RAG 必须只检索当前用户的 `ready` 文档。

当前列使用不固定维度的 `VECTOR`，因为尚未确定 Embedding 模型：

- 确定模型前不要混合写入不同维度的生产向量；
- 选定模型后通过新迁移改为 `VECTOR(n)`；
- 维度固定后再创建 HNSW 索引；
- 保存 `embedding_model`，方便模型升级和重建；
- 更换模型通常需要重新生成全部向量。

小数据量可以精确检索，HNSW 是数据增长后的性能优化。

## 十三、学习目标表 `learning_goal`

### 13.1 职责

保存长期学习目标，例如“掌握 Spring Security”或“完成 DeveloperOS MVP”。

### 13.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 目标唯一 ID |
| `user_id` | `BIGINT` | 必填，外键 | 所属用户 |
| `title` | `VARCHAR(200)` | 必填 | 目标标题 |
| `description` | `TEXT` | 可空 | 背景、范围、完成标准 |
| `status` | `VARCHAR(20)` | 默认 `active` | 目标状态 |
| `priority` | `SMALLINT` | 默认 3 | 优先级 1 到 5，越大越重要 |
| `progress_percent` | `SMALLINT` | 默认 0 | 进度 0 到 100 |
| `start_date` | `DATE` | 可空 | 计划开始日期 |
| `target_date` | `DATE` | 可空 | 计划完成日期 |
| `completed_at` | `TIMESTAMPTZ` | 可空 | 实际完成时间 |
| `created_at` | `TIMESTAMPTZ` | 自动 | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 自动 | 最后修改时间 |

状态：`draft`、`active`、`paused`、`completed`、`cancelled`。

数据库保证目标日期不早于开始日期，进度在 0 到 100，优先级在 1 到 5。

V1 推荐进度规则：

```text
已完成且未取消任务数 / 未取消任务总数 * 100
```

任务状态变化后，Growth Service 在同一事务重算进度。没有任务时保持 0；目标完成时设为 100 并填写 `completed_at`。

`UNIQUE (id, user_id)` 虽然从普通唯一性看冗余，但它是 `growth_log` 复合外键的被引用键，用于保证日志只能关联同一用户的目标。

## 十四、学习任务表 `learning_task`

### 14.1 职责

把目标拆成可执行任务，也用于每日任务列表。

### 14.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 任务唯一 ID |
| `goal_id` | `BIGINT` | 必填，外键 | 所属目标 |
| `title` | `VARCHAR(200)` | 必填 | 任务标题 |
| `description` | `TEXT` | 可空 | 任务说明和完成标准 |
| `status` | `VARCHAR(20)` | 默认 `todo` | 任务状态 |
| `priority` | `SMALLINT` | 默认 3 | 优先级 1 到 5 |
| `scheduled_date` | `DATE` | 可空 | 计划哪一天执行 |
| `due_at` | `TIMESTAMPTZ` | 可空 | 最晚完成时间 |
| `sort_order` | `INTEGER` | 默认 0 | 同一目标内排序值 |

## 十五、成长日志表 `growth_log`

### 15.1 职责

保存每日、每周、里程碑和复盘记录，形成长期成长轨迹。

### 15.2 字段

| 字段 | 类型 | 必填/默认 | 作用 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 主键 | 日志唯一 ID |
| `user_id` | `BIGINT` | 必填，外键 | 所属用户 |
| `goal_id` | `BIGINT` | 可空，复合外键一部分 | 可选关联学习目标 |
| `log_type` | `VARCHAR(20)` | 默认 `daily` | 日志类型 |
| `title` | `VARCHAR(200)` | 必填 | 日志标题 |
| `content` | `TEXT` | 必填 | 日志正文 |
| `log_date` | `DATE` | 默认当天 | 日志记录的业务日期 |
| `duration_minutes` | `INTEGER` | 可空 | 学习或实践分钟数 |
| `is_ai_generated` | `BOOLEAN` | 默认 `false` | 是否由 AI 生成或总结 |
| `metadata` | `JSONB` | 默认空对象 | 标签、情绪、引用任务等 |

| `updated_at` | `TIMESTAMPTZ` | 自动 | 最后修改时间 |

日志可以不属于任何目标，所以 `goal_id` 可空。复合外键：

```sql
FOREIGN KEY (goal_id, user_id)
REFERENCES learning_goal (id, user_id)
```

它阻止用户 A 的日志关联用户 B 的目标。删除目标时只把日志的 `goal_id` 置空，日志本身保留，因为成长历史不能随目标删除而消失。

## 十六、关系和删除规则

| 父表 | 子表 | 关系 | 删除父记录行为 | 原因 |
| --- | --- | --- | --- | --- |

| `conversation` | `message` | 一对多 | 删除消息 | 消息脱离会话无意义 |

## 十七、核心业务数据流

### 17.1 注册和登录

```text
DTO 校验
 -> 标准化 username/email
 -> 哈希密码
 -> 写入 app_user
 -> 登录成功更新 last_login_at
 -> Token 状态写入 Redis
```

数据库永远不保存明文密码。密码哈希不可解密，只能通过安全算法验证。

### 17.2 AI 对话

```text
创建 conversation
 -> 写入 user message
 -> 更新 last_message_at
 -> 调用 LLM，通过 SSE 输出
 -> 写入或更新 assistant message
 -> 更新 message.status
 -> 必要时提取 memory
```

数据库操作需要明确事务边界，外部 LLM 调用不应长时间占用数据库事务。

### 17.3 长期记忆

```text
读取近期 conversation/message
 -> AI 判断长期价值
 -> 去重和冲突检查
 -> 写入或更新 memory
 -> 后续对话检索 active memory
 -> 组装到 Prompt
```

不能把所有聊天消息都直接变成记忆，否则会产生重复、冲突和无效上下文。

### 17.4 RAG 文档处理

```text
上传文件到文件存储
 -> 创建 document(pending)
 -> 修改为 processing
 -> 解析和切片
 -> 生成 Embedding
 -> 保存 chunk
 -> 更新 chunk_count
 -> 修改 document 为 ready
```

问答时只检索当前用户 `ready` 文档的切片，取 Top K 加入 Prompt，并向用户展示来源。

### 17.5 学习成长闭环

```text
创建 learning_goal
 -> 拆分 learning_task
 -> 展示每日任务
 -> 更新任务状态
 -> 重算 progress_percent
 -> 写入 growth_log
 -> AI 生成复盘
```

对应产品闭环：

```text
学习 -> 实践 -> 沉淀 -> 复盘 -> 再成长
```

## 二十、用户数据隔离和安全

### 20.1 所有权校验

外键不能识别当前请求用户。所有查询和修改都必须验证当前登录用户：

- `message -> conversation -> user_id`；
- `learning_task -> learning_goal -> user_id`；
- `knowledge_chunk -> knowledge_document -> user_id`；
- `growth_log.user_id`；
- `memory.user_id`。

不能只按子资源 ID 查询后直接返回。

### 20.2 密码和 Token

- 密码只保存安全哈希；
- JWT 不包含密码或密钥；
- Token 黑名单、刷新状态或会话状态存 Redis；
- 日志不能打印密码、Token 和敏感 Prompt。

### 20.3 文件安全

- 文件操作都校验用户；
- 存储 Key 由后端生成；
- 限制文件大小、类型和解析时间；
- 不可信文档内容不能直接变成系统级 Prompt 指令。

### 20.4 备份

长期使用前必须备份 PostgreSQL 和文件存储。只备份数据库而没有备份原文件，知识库无法完整恢复。

## 二十一、JSONB 和向量使用规则

### 21.1 JSONB

当前 JSONB 字段包括用户偏好以及消息、记忆、文档、切片、日志的 `metadata`。

适合 JSONB：结构可能变化、很少筛选排序、不需要唯一约束的附加信息。

不适合 JSONB：用户 ID、状态、标题、时间、需要关联和约束的核心字段。

如果某个 JSONB 属性开始频繁查询，应通过迁移提升为普通列。MyBatis-Plus 需要明确 JSONB TypeHandler。

### 21.2 向量

正式写入向量前必须确定：

- Embedding 模型；
- 输出维度；
- 相似度算法；
- 切片规则；
- 模型升级后的重建方式。

确定后使用新迁移固定 `VECTOR(n)` 并创建 HNSW 索引，不能靠重新执行 V1 脚本升级字段。

## 二十二、数据库迁移规范

### 22.1 为什么需要 Flyway

`CREATE TABLE IF NOT EXISTS` 只能首次创建，不能升级已有表。

长期项目应使用：

```text
V1__init_schema.sql
V2__fix_embedding_dimension.sql
V3__add_memory_embedding.sql
```

Flyway 会维护迁移历史，让本地、Docker、测试和部署环境按相同顺序升级。

### 22.2 基本规则

1. 当前 `V1__init_schema.sql` 是正式初始迁移；
2. V1 一旦执行，不再修改；
3. 后续字段、索引和约束变化都新增迁移；
4. 不用手工改表代替迁移；
5. 重要迁移前备份；
6. 先在测试数据库验证；
7. 不依赖 `IF NOT EXISTS` 掩盖环境差异。

PostgreSQL 支持事务型 DDL，Flyway 会管理迁移事务，因此迁移文件中不再手工编写 `BEGIN` 和 `COMMIT`。迁移失败时不会写入成功记录。

## 二十七、常见问题

### 27.1 为什么不用 MySQL？

PostgreSQL 对 JSONB、复杂查询、约束和扩展能力支持较好，pgvector 能直接满足 RAG 场景。

### 27.2 为什么不一开始使用微服务？

模块化单体更容易开发、调试、部署和保证事务。等业务和团队规模真正需要时再拆分。

### 27.3 为什么状态不用 PostgreSQL ENUM？

`VARCHAR + CHECK` 更容易通过迁移增加状态，也容易和 Java Enum 配合。

### 27.4 为什么有些子表没有 `user_id`？

它们可以通过父表确定用户，避免重复和不一致。后端必须通过父表校验所有权。

### 27.5 为什么记忆不等于聊天记录？

聊天包含大量临时内容。长期记忆必须经过提取、去重、冲突处理和重要性判断。

### 27.6 为什么日志同时有 `user_id` 和 `goal_id`？

日志可以不属于目标，所以直接保存用户；有关联目标时，复合外键保证目标属于同一用户。

### 27.7 为什么不把文件直接存数据库？

大文件会增加数据库备份、恢复和查询压力。数据库保存元数据，文件交给文件系统或对象存储。

### 27.8 初始化脚本可以重复执行吗？

部分 `IF NOT EXISTS` 可以避免重复创建，但它不是升级工具，不能代替 Flyway。正式环境不能靠反复执行 V1 升级结构。

## 二十八、V1 数据库完成标准

创建表不等于数据库模块完成。至少应满足：

- 所有迁移可以从空数据库稳定执行；
- Entity、Mapper 和字段一致；
- 关键 Service 使用事务；
- 用户数据隔离有集成测试；
- 九张表的基本业务链路可以运行；
- 删除和级联行为有测试；
- RAG 只返回当前用户数据；
- 数据库和文件可以备份恢复；
- Docker 和本地使用同一套迁移；
- 文档和迁移脚本保持同步。

满足这些条件后，数据库才真正成为 DeveloperOS 可长期维护的基础，而不只是创建了几张表。
